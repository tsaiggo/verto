# Component Library

Beyond the standard Markdown blocks, Verto ships a small library of MDX components designed for documentation-style content.

## Tabs

Use `<Tabs>` to group related variants — package managers, OS flavours, code dialects.

<Tabs id="install">
  <Tab label="npm">
    ```bash
    npm install verto
    ```
  </Tab>
  <Tab label="pnpm">
    ```bash
    pnpm add verto
    ```
  </Tab>
  <Tab label="yarn">
    ```bash
    yarn add verto
    ```
  </Tab>
</Tabs>

When `id="install"` is set, the active tab syncs with the URL hash — share `…#install=pnpm` and your reader lands on the right tab. Arrow keys navigate the tablist for accessibility.

## Steps

`<Steps>` numbers heading-led blocks automatically. Pure CSS counter, zero JS:

<Steps>
### Drop in your files

Put any `.md` or `.mdx` files under `content/`.

### Run the dev server

```bash
npm run dev
```

### Open `/read`

Verto auto-discovers everything and serves it under the reader.
</Steps>

## Accordion

Single panel, or grouped — pass `exclusive` to enforce one-open-at-a-time:

<AccordionGroup exclusive>
  <Accordion title="Why footnote-style inline comments?">
    Because `[^c-id]` is valid CommonMark syntax. GitHub, Typora, and any other Markdown renderer fall back to a regular footnote — your annotations are never lost.
  </Accordion>
  <Accordion title="Does Verto need a database?">
    No. Verto reads from the filesystem at build time. The output is fully static.
  </Accordion>
  <Accordion title="Can I customize the navigation order?">
    Yes — drop a `content/navigation.json` with `overrides` keyed by slug. See the docs on the file tree.
  </Accordion>
</AccordionGroup>
