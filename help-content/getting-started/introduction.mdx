# Welcome to Verto

Verto is a reader-first knowledge workspace for Markdown and MDX. Point it at
files you control, then read, search, organize, annotate, and optionally ask a
grounded Agent about them from one Codex-style workspace.

## Why Verto?

Folders are the library. Verto does not require a database, a CMS, or
proprietary document formats. Titles, navigation, tags, and reading views are
derived from ordinary `.md` and `.mdx` files, with optional frontmatter when
you want more control.

The browser experience focuses on reading. The desktop app can also connect a
local folder, edit its Markdown or MDX files, and save changes back to that
folder. Your files remain portable to editors such as VS Code, Obsidian,
Cursor, or vim.

<Callout type="info">
  The workspace uses a **Codex-style shell**: persistent navigation on wide
  screens, a focused task surface in the middle, and an integrated Contents or
  Companion inspector when a document has enough context to need one.
</Callout>

## What works today

- **Library and Reader** — browse files, search headings and code, follow tags,
  track recent documents, and read with a generated table of contents.
- **Reading organization** — bookmark documents, collect them into named sets,
  and keep reading state and annotations beside the library in desktop builds.
- **Rich Markdown and MDX** — callouts, toggles, task lists, figures, tables,
  diagrams, highlighted code, and inline author comments render in the Reader.
- **Local and RSS runtime sources** — connect a local Markdown/MDX folder to the
  Library, or subscribe to RSS and Atom feeds that arrive in Inbox.
- **Portable writing** — preview a Markdown or MDX draft in the Reader type
  system. Web builds export `.mdx`; desktop builds can save into a connected
  local library.
- **Optional grounded Agent** — supported builds can use GitHub Models after
  you add a token on the current device. Verto does not require AI for reading,
  writing, search, or organization.

## Where content comes from

Verto has two source paths with different update behavior:

1. **Runtime workspace sources.** A local folder becomes the live Library.
   RSS and Atom subscriptions are kept separately in Inbox.
2. **Build-time sources.** Static deployments can read a local directory or a
   OneDrive folder. A build-time source change appears after a rebuild.

Bundled Help always comes from `help-content/`; switching your Library does not
replace the product documentation.

<Callout type="warning">
  **Capability limits:** the web app depends on browser support for folder
  access and keeps workspace state in `localStorage`. OneDrive is a build-time
  source, not a live desktop sync provider. The Agent is available only when a
  supported build enables GitHub Models and the current device has a token;
  development builds may expose deterministic demo responses instead.
</Callout>

## Principles

1. **Files stay yours.** Markdown and MDX remain the source of truth.
2. **Reading works without AI.** Core navigation, search, writing, and
   organization do not require an assistant.
3. **State is inspectable.** Desktop libraries store portable workspace state
   in a hidden `.verto/` directory; web-only state remains on the device.
4. **Assistant actions are explicit.** Verto sends context only after you ask
   and discloses when a request uses partial source context.
