# Responsive Design

Verto keeps the same information architecture across window sizes, but it does
not force the desktop rail and inspector into narrow layouts. The workspace
moves navigation and secondary tools out of the way before the document column
becomes cramped.

## Layout ranges

| Range | Workspace behavior |
|-------|--------------------|
| Narrow (below 1024px) | The persistent rail is removed, the work surface uses the viewport, and the top bar exposes a navigation trigger. |
| Desktop (1024px and above) | The Codex-style rail stays visible beside the rounded work surface. Its width is fluid rather than a fixed legacy sidebar size. |
| Wide Reader (1280px and above) | A Contents inspector can sit beside the document. Opening Reading Companion reuses that inspector instead of adding another competing rail. |
| Wide Agent (above 1360px) | Agent context can appear as a floating inspector while the conversation remains the primary surface. |

These boundaries control layout behavior, not what content is available.
Library, Search, Settings, Help, and Reader keep the same routes and actions
when the shell changes shape.

<Callout type="info">
  Verto reserves the widest multi-column layouts for windows that can keep a
  readable document column. A hidden rail or inspector does not leave an empty
  grid column behind.
</Callout>

## Navigation on narrow screens

Below the desktop range, the menu button opens the same primary rail inside an
accessible modal drawer. It is not a separate list of mobile-only links.

The drawer:

- keeps the normal Library, Inbox, Sources, Studio, Agent, project, and document
  navigation;
- traps focus while it is open and supports Escape and an explicit close
  button;
- closes after navigation and when the window returns to the desktop range;
- restores focus to the menu trigger, or to the main content when the trigger
  is no longer available; and
- uses a backdrop so the navigation state is clear without replacing the whole
  page.

The top bar condenses secondary actions and truncates long document identities
instead of letting them push the layout wider than the viewport.

## Reader and Companion

The Reader centers a bounded text column and lets Reading settings change its
width, density, text size, and typeface. A document can opt out of its generated
table of contents with frontmatter, but the document body remains unchanged.

On wide screens, Contents lives in one card beside the article. Reading
Companion opens in that same card and provides a Contents control to return.
This avoids the old design where TOC and Companion competed for separate right
rails.

On smaller screens, Verto does not reserve a permanent inspector column.
Document actions stay near the Reader header, and secondary tools open from
their compact controls.

## Content that reflows

- Tables and code blocks scroll within their own containers instead of widening
  the page.
- Library and settings layouts collapse secondary columns before the primary
  task becomes too narrow.
- Search moves filters into a compact sheet on smaller screens.
- Collection selection can use a bottom sheet in narrow Reader layouts.
- Bookmark and other row actions remain visible on touch devices that cannot
  hover.
- Theme and Reader preferences persist while the layout changes.

## Accessibility behavior

The shell includes a skip-to-content link. The navigation drawer uses dialog
semantics and managed focus. Visible focus styles remain on TOC links and
controls, and Verto removes nonessential transitions when the operating system
requests reduced motion.

<Callout type="warning">
  Responsive browser previews cannot verify native desktop behavior. Local
  folder selection and direct file saving still depend on the desktop runtime
  or on browser support for the File System Access API.
</Callout>
