# Build and Runtime Content

Verto supports two content paths. A configured **build-time source** is
pre-rendered into the application, while a **runtime local library** is chosen
from Sources and opened on the current device. They use the same Reader design,
but they have different update and storage behavior.

## Build-time routes

The web and desktop builds pre-render two document trees:

- `app/read/[[...path]]/page.tsx` enumerates the configured Library source and
  emits `/read` plus every readable Library path.
- `app/help/[[...path]]/page.tsx` enumerates bundled `help-content/` and emits
  `/help` plus every Help path.

Both routes use `generateStaticParams()`. During the build, Verto parses
frontmatter, builds the navigation tree, compiles MDX on the server, extracts
the table of contents, and renders Shiki-highlighted code into the page markup.
The browser does not compile those source files again.

<Callout type="info">
  Help is pinned to `help-content/`. Changing the Library source never replaces
  product Help.
</Callout>

## Choose the build source

The default source is the repository's `content/` directory. A different local
directory can be selected before a build:

```bash
VERTO_CONTENT_SOURCE=local
VERTO_LOCAL_DIR=/path/to/library
npm run build
```

Static deployments can instead use the OneDrive source configured through the
`VERTO_ONEDRIVE_*` environment variables. OneDrive is a build adapter, not a
live sync connection in Sources.

Changing files in either build-time source requires another build before the
deployed `/read/*` pages change.

## Web output

```bash
npm run build
npm start
```

The normal Next.js build writes `.next/`. `npm start` serves that production
application, including its pre-rendered pages and client-side navigation. It is
not a standalone folder of HTML that should be served without Next.js.

A Next.js platform such as Vercel can build and run it. Configure any content
source environment variables in the deployment's build environment.

## Desktop static export

The desktop build sets `TAURI=1`, which enables Next.js `output: "export"` and
writes a static `out/` directory for the Tauri shell:

```bash
npm run build:tauri
```

Tauri loads that export from disk, so no Node server is required when the
desktop application runs. Legacy web redirects are intentionally omitted from
this export because Next.js static export cannot emit them.

## Runtime local libraries

**Sources → Local Library** follows a different path and does not call
`generateStaticParams()`:

- In a supported browser, Verto asks for a directory, reads its Markdown and
  MDX files, and caches their text and index on the current device.
- In the desktop app, Tauri authorizes the selected directory and reads it
  through native commands.
- Documents open in the runtime Reader route at `/runtime/local`; selecting a
  new runtime folder updates Library and Explorer without rebuilding Verto.

The pre-rendered `/read/*` tree still reflects the source used for the build.
Connecting a runtime folder changes the live workspace Library; it does not
rewrite deployed route files or upload the selected folder anywhere.

| Goal | Use | Rebuild? |
|------|-----|----------|
| Ship a fixed Library with the app or site | Local or OneDrive build source | Yes, after source changes |
| Read a folder on this browser | Browser local connection | No; choose again to refresh the cache |
| Read and edit a folder in Verto desktop | Desktop local connection | No |
| Keep product documentation available | Bundled Help source | Only when Help itself changes |
